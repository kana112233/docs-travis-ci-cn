通过发送POST请求，使用API​​ V3构建Trigger Travis CI/repo/{slug|id}/requests：

  1. 从Travis CI获取API令牌个人资料页。您需要令牌来验证大多数这些API请求。 您也可以使用Travis CI命令行客户端

获取您的API令牌：

travis token --com

关于哪些端点可用以及您可以使用它们做什么。

If your API or cron triggered builds run on the same commit or branch over and over again, you will notice GitHub have a limit of 1000 status updates per commit, which causes the Travis CI build status displayed on GitHub to be incorrect. Avoid this problem by specifying a regularly updated branch for your automated builds.

自定义提交消息 #

您可以在请求正文中指定提交消息：

body='{
  "request": {
    "branch":"master",
    "message": "Override the commit message: this is an api request"
  }
}'

自定义构建配置 #

您还可以自定义构建配置。

选择以下合并模式之一以确定原始构建配置的方式（在.travis.yml文件），并且组合了API请求中指定的配置：

  • replace使用API​​请求正文中的配置替换完整的原始构建配置
  • merge（默认）使用API​​请求正文中的配置替换原始配置中的部分
  • deep_merge将API请求正文中的配置合并到构建配置中

例如，鉴于此.travis.ymlfile包含以下配置：

language: ruby
env:
  FOO: foo_from_travis_yaml

{：data-file =“。travis.yml”}

鉴于我们将以下配置作为API请求发送：

body='{
  "request": {
    "config": {
      "merge_mode": [...],
      "env": {
        "BAZ": "baz_from_api_request"
      },
      "script": "echo FOO"
    }
  }
}'

合并模式：替换 #

随着merge_mode调成replace生成的组合构建配置是：

{
  "env": {
    "BAZ": "baz_from_api_request"
  },
  "script": "echo FOO"
}

完整配置已由API请求中的配置替换。

合并模式：合并（默认） #

随着merge_mode调成merge，或者没有给出merge_mode（默认），生成的构建配置是：

{
  "language": "ruby"
  "env": {
    "BAZ": "baz_from_api_request"
  },
  "script": "echo FOO"
}

每个顶级部分都被替换为

API请求。其他顶级部分保持不变。

合并模式：deep_merge #

随着merge_mode调成deep_merge，生成的构建配置是：

{
  "language": "ruby"
  "env": {
    "FOO": "foo_from_travis_yaml",
    "BAZ": "baz_from_api_request"
  },
  "script": "echo FOO"
}

该envAPI请求中的部分已合并到现有的env部分。
