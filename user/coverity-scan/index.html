Using Coverity Scan with Travis CI

  • 什么是静态分析？
  • 建立提交频率
  • macOS支持
  • 逐步配置
  • 环境变量

Coverity扫描是一个免费的静态代码分析工具，适用于Java，C，C ++和C＃。它分析每行代码和潜在的执行路径，并生成潜在代码缺陷列表。通过使用Coverity Scan增强CI流程，您可以进一步了解代码的质量，超出自动化测试所涵盖的范围。

此插件利用Travis CI基础架构自动在GitHub项目上运行代码分析。

什么是静态分析？ #

静态分析是一组用于查找源代码缺陷和漏洞的过程。

在静态分析中，不执行检查中的代码。因此，不需要测试用例和特殊设计的输入数据集。对缺陷和漏洞的检查不仅限于在一些代码执行期间运行的代码行，还可以包括代码库中的所有代码行。

此外，Coverity的静态分析实现可以通过源代码（包括过程间）跟踪所有可能的执行路径，并查找由非错误彼此独立的语句组合引起的缺陷和漏洞。

有关Coverity Scan的详细信息，请参阅常问问题。

建立提交频率 #

在项目的每次提交中运行静态分析可能有点过分。为了提高免费服务对更多项目的可用性，默认情况下会设计插件以按分支运行分析。我们建议您创建一个名为的分支coverity_scan，您可以在想要触发分析时合并到其中。见常问问题有关构建提交频率的信息。

macOS支持 #

Coverity Scan插件不适用于启用了SIP功能的macOS版本，即在macOS El Capitan（10.11）及更高版本上。特别是在Travis CI上，它目前仅适用于我们的Xcode 6.4图像（即带有osx_image: xcode6.4）。但是，可以使用自定义脚本或命令。请联系support@travis-ci.com学习如何。

逐步配置 #

  1. 注册使用您的GitHub帐户进行Coverity Scan（如果您还没有）。
  2. 如有必要，请创建一个公共回购GitHub上为您的项目。
  3. 如有需要，请注册特拉维斯CI并按照以下内容配置项目获得教程指南。
  4. 登录以扫描，然后添加您的项目。一定要把它添加为GitHub项目。
  5. （可选）首次对项目使用Coverity Scan时，您可能希望在自己的开发计算机上进行构建，以确保一切正常完成。这是可选的，但它可以简化任何必要的调试。咨询Coverity Scan下载页面作为指示。
  6. 从Coverity扫描仪表板中，单击Project Settings。然后，在右侧，单击Submit build按钮。
  7. 假设项目已通过GitHub正确注册，您将看到一个标签Configure Travis CI。访问该小组。
  8. 在Travis CI配置页面上，您将看到一个示例.travis-yml文件。它将自动生成几个必要的项目特定字段，包括加密的Coverity Scan令牌（上传结果所必需的）。将配置从那里复制到您的.travis.yml文件，在接下来的步骤中进行更改。
  9. 填写build_command_prepend使用准备构建项目所需的任何配置命令。例如build_command_prepend: ./configure
  10. 填写build_command使用命令来构建项目。这将作为参数提供给cov-build命令。例如：build_command: make
  11. 将这些更改提交给coverity_scan科。
  12. 推动coverity_scan分支到GitHub。等待Travis CI查看提交，然后开始构建。
  13. 访问特拉维斯CI直接，或点击你的按钮Project Settings页面，将在Travis CI上激活项目后显示。

travis.yml #

从Coverity Scan上的项目页面中，选择Travis CI选项卡。你会看到一段YAML被复制到你的.travis-ci文件。请注意，这是一个示例，可能需要进行一些调整才能使构建正常运行。

env:
  global:
    # COVERITY_SCAN_TOKEN
    # ** specific to your project **
    - secure: "xxxx"

addons:
  coverity_scan:

    # GitHub project metadata
    # ** specific to your project **
    project:
      name: my_github/my_project
      version: 1.0
      description: My Project

    # Where email notification of build analysis results will be sent
    notification_email: scan_notifications@example.com

    # Commands to prepare for build_command
    # ** likely specific to your build **
    build_command_prepend: ./configure

    # The command that will be added as an argument to "cov-build" to compile your project for analysis,
    # ** likely specific to your build **
    build_command: make

    # Pattern to match selecting branches that will run analysis. We recommend leaving this set to 'coverity_scan'.
    # Take care in resource usage, and consider the build frequency allowances per
    #   https://scan.coverity.com/faq#frequency
    branch_pattern: coverity_scan

{：data-file =“。travis.yml”}

项目设置应该是不言自明的，并且应该与Coverity Scan上项目配置的值相匹配。该branch_pattern是要运行Coverity Scan的分支的正则表达式。请参考常问问题关于在启用其他分支之前的构建提交限制我们建议将此设置保留coverity_scan。

COVERITY_SCAN_TOKEN已加密，可通过使用获取Travis CI CLI。为方便起见，Coverity Scan在项目的Travis CI选项卡上提供此信息，但您也可以手动运行（请参阅加密密钥有关加密的更多信息）。

gem install travis
cd my_project
travis encrypt COVERITY_SCAN_TOKEN=project_token_from_coverity_scan

然后复制生成的行，如YAML示例中所示。

环境变量 #

定义后，以下环境变量将覆盖它们

相应的配置值.travis.yml

  1. COVERITY_SCAN_NOTIFICATION_EMAIL
  2. COVERITY_SCAN_BUILD_COMMAND
  3. COVERITY_SCAN_BUILD_COMMAND_PREPEND
  4. COVERITY_SCAN_BRANCH_PATTERN

##

禁用后续测试运行 #

由于Travis CI插件的运行方式，您的标准脚本阶段（即测试）将在Coverity Scan分析完成后运行。为了避免这种情况，您可以修改您的script指令.travis.yml。

该COVERITY_SCAN_BRANCH环境变量将设置为1当Coverity Scan插件正在运行时。因此，您可以更改脚本

script: make

{：data-file =“。travis.yml”}

至

script: if [ ${COVERITY_SCAN_BRANCH} != 1 ]; then make ; fi

一定要更换make使用标准CI构建命令。
