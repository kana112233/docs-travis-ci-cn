Caching Dependencies and Directories

  • 缓存和读取权限

Travis CI可以缓存不经常更改的内容，从而加快构建过程。

要使用缓存功能，在您的存储库设置中，设置建立推动分支至

上。

  • Travis CI为每个构建提取缓存，包括分支和拉取请求。
  • 如果分支没有自己的缓存，Travis CI将获取存储库默认分支的缓存。
  • 每个分支有一个缓存和语言版本/编译器版本/ JDK版本/ Gemfile位置/等。
  • 仅存储从正常推送对缓存目录所做的修改。

请注意，缓存内容可用于存储库上的任何构建，包括Pull Requests，因此请确保不要在缓存中放置任何敏感信息。

##

缓存允许Travis CI在构建之间存储目录，这对于存储很有用

需要更长时间才能编译或下载的依赖项。

请注意，如果第三方项目（如Bundler）更改了存储依赖项的位置，则可能需要指定联系我们有任何问题，问题或反馈。

捆绑 #

可以构成构建持续时间的很大一部分。在构建之间缓存bundle会大大减少构建运行所需的时间。

启用Bundler缓存 #

.travis.yml

language: ruby

每当您更新捆绑包时，Travis CI也会更新缓存。

确定捆绑路径 #

Travis CI尽力确定路径捆绑器用于存储依赖关系。

如果你有自定义Bundler参数，这些包括--path选项，Travis CI将使用该路径。如果--path不见了但是--deployment存在，它将使用vendor/bundle。

否则会自动添加--path选项。在这种情况下，它将使用环境变量的值BUNDLE_PATH或者，如果遗漏，vendor/bundle。

的CocoaPods #

Git存储库中的Pods目录。

拉请求构建和缓存 #

Pull请求构建按顺序检查以下缓存位置，使用第一个存在位置：

  • 拉请求缓存。
  • 拉请求目标分支缓存。

如果以前的任何位置都不包含有效的缓存，则构建将继续，而不会缓存。

在第一次拉取请求构建运行后，它会创建一个新的拉取请求缓存。

关于拉取请求的缓存需要注意的一些重要事项：

  • 如果存储库有建立推动分支调成关闭，目标分支和主分支都不能被缓存。
  • 如果主分支上的缓存是旧的，例如在大多数工作发生在分支上的工作流中，缓存的用处就越少。
  • 如果拉取请求正在使用缓存但您不希望它，则需要清除都拉请求缓存和目标分支的缓存。

before_cache阶段 #

使用缓存时，在上载之前运行命令可能很有用

新的缓存存档。

例如，依赖关系管理实用程序可能会将日志文件写入您正在缓存的目录中，并且您不希望它们影响缓存。使用before_cache阶段删除日志文件：

cache:
  directories:
    - $HOME/.cache/pip
before_cache:
  - rm -f $HOME/.cache/pip/log/debug.log

{：data-file =“。travis.yml”}

此阶段的失败并未将作业标记为失败。

清除缓存 #

有时，您通过将错误数据存储在其中一个缓存目录中来破坏缓存，或者当语言运行时更改时缓存可能会变为无效。

使用以下方法之一访问缓存并在必要时将其删除：

  • 您的存储库的设置页面https://travis-ci.com
  • 该命令行客户端
  • 该API

请注意，如果您仍在使用travis-ci.org您需要使用.org网址来访问您的设置页面和API请求。

缓存过期 #

缓存存档目前设置为在开源项目28天后到期，私有项目为45天。这意味着如果特定缓存存档在到期延迟后未更改，则将删除该缓存存档。

组态 #

启用多个缓存功能 #

如果要启用多个缓存功能并且语言支持它们，可以将它们列为数组：

language: objective-c
cache:
- bundler
- cocoapods

{：data-file =“。travis.yml”}

缓存时这不起作用任意目录，

或者当语言不支持任何指令时。

如果要将其与其他缓存模式结合使用，请使用哈希映射。

以下是一个Ruby存储库缓存Node.js模块的示例：

language: ruby
cache:
  bundler: true
  directories:
  - node_modules # NPM packages

{：data-file =“。travis.yml”}

这是另一个例子;一个Rust存储库缓存货物和Ruby宝石：

language: rust
cache:
  cargo: true
  directories:
    - vendor/bundle
install:
  - bundle install --deployment # to cache vendor/bundle

{：data-file =“。travis.yml”}

明确禁用缓存 #

您可以通过设置显式禁用所有缓存cache选项false在你的.travis.yml：

cache: false

{：data-file =“。travis.yml”}

也可以禁用单个缓存模式：

language: objective-c
cache:
  bundler: false
  cocoapods: true

{：data-file =“。travis.yml”}

设置超时 #

默认情况下，缓存的超时设置为3分钟。超时是有序的

防止任何可能导致构建卡住的问题。这样的问题可能是

由工作服务器和S3之间的网络问题引起的，甚至是由缓存引起的

太大，无法及时打包和上传。但是，有

您可能希望设置更大的超时的情况，尤其是在您需要时

缓存大量。为了更改超时，您可以使用timeout

具有所需时间的财产：

cache:
  timeout: 1000

{：data-file =“。travis.yml”}

缓存和构建矩阵 #

当你有一个多个工作构建矩阵，

每个作业的一些特征用于识别每个作业的缓存

工作应该使用。

这些因素是：

  1. 操作系统名称（目前，linux，osx， 要么windows）
  2. 操作系统分发（适用于Linux，xenial，trusty， 要么precise）
  3. macOS图像名称（例如，xcode7.2）
  4. 设置的可见环境变量的名称和值.travis.yml或“设置”面板
  5. 语言运行时版本（用于语言中指定的语言）language密钥）如果适用的话
  6. 对于Bundler感知作业，名称Gemfile用过的

如果这些特征由构建矩阵中的多个作业共享，

他们将在网络上共享相同的URL。

这可能会破坏缓存，或者缓存可能包含不缓存的文件

CACHE_NAME=JOB1

{：data-file =“。travis.yml”}

究竟，

包括空格。

例如，用

env:
  - FOO=1 BAR=2
  - FOO=1  BAR=2
  - BAR=2 FOO=1

{：data-file =“。travis.yml”}

三个作业中的每一个都将使用自己的缓存。

缓存和读取权限 #

缓存时自定义文件和目录，

确保您指定的位置是用户可读写的。

如果不是，则缓存实用程序会报告错误

所调用tar

FAILED: tar -Pzcf /Users/travis/.casher/push.tgz /path/to/unreadable/directory

tar: /path/to/unreadable/directory: Cannot stat: No such file or directory

缓存如何工作？ #

Travis CI会保存配置和上传中列出的所有目录的存档

它使用安全且受保护的URL到存储提供商，确保安全和隐私

上传的档案。

请注意，这使得我们的缓存不是网络本地的，它仍然绑定到网络

在缓存中。如果您存储大于几百兆的存档

缓存，你不太可能看到一个很大的速度提升。

在构建之前，我们检查是否存在缓存存档。如果是，我们下载并将其解压缩到指定位置。

在构建之后，我们检查目录中的更改，创建包含这些更改的新存档，然后将其上载到远程存储。

上传目前是构建周期的一部分，但我们正在寻求改进

这种情况发生在构建之外，提供更快的构建反馈。
