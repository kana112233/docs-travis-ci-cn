本指南涵盖的内容 #

本指南介绍了使用Travis提供的工具进行无头GUI和浏览器测试CI环境。大多数内容都是技术中立的，并未涵盖特定测试工具的所有细节（如Poltergeist或Capybara）。我们建议您先从教程和构建配置阅读本文之前的指南。

使用酱汁实验室 #

酱汁实验室提供Selenium云，可访问170多种不同的设备/操作系统/浏览器组合。如果你有使用Selenium的浏览器测试，使用Sauce Labs来运行测试非常容易。首先，您需要注册他们的服务（它对于开源项目是免费的）。

注册后，使用Sauce Connect设置隧道，以便Sauce Labs可以连接到您的Web服务器。我们的酱连接插件使这很简单，只需将其添加到.travis.yml：

addons:
  sauce_connect:
    username: "Your Sauce Labs username"
    access_key: "Your Sauce Labs access key"

{：data-file =“。travis.yml”}

您可以加密您的访问密钥， 如果你想。

现在，Sauce Labs有一种方法可以访问您的Web服务器，但您仍然需要启动它。看到启动Web服务器以下是有关如何执行此操作的详细信息。

最后，您需要将Selenium测试配置为在Sauce Labs上运行，而不是在本地运行。这是使用a完成的远程WebDriver。确切的代码取决于您使用的工具/平台，但对于Python，它看起来像这样：

username = os.environ["SAUCE_USERNAME"]
access_key = os.environ["SAUCE_ACCESS_KEY"]
capabilities["tunnel-identifier"] = os.environ["TRAVIS_JOB_NUMBER"]
hub_url = "%s:%s@localhost:4445" % (username, access_key)
driver = webdriver.Remote(desired_capabilities=capabilities, command_executor="http://%s/wd/hub" % hub_url)

Sauce Connect插件出口了SAUCE_USERNAME和SAUCE_ACCESS_KEY环境变量，并将与中心URL的连接中继回Sauce Labs。

这就是您在Sauce Labs上运行Selenium测试所需的全部内容。但是，您可能只想将Sauce Lab用于Travis CI构建，而不是用于本地构建。为此，您可以使用CI要么TRAVIS环境变量有条件地改变你正在使用的驱动程序（见我们的可用环境变量列表有关更多方法来检测您是否在Travis CI上运行）。

为了使Sauce Labs上的测试结果更容易导航，您可能希望提供更多元数据以与构建一起发送。您可以通过传递更多所需功能来实现此目的：

capabilities["build"] = os.environ["TRAVIS_BUILD_NUMBER"]
capabilities["tags"] = [os.environ["TRAVIS_PYTHON_VERSION"], "CI"]

对于travis-web，我们自己的网站，我们使用Sauce Labs在多个浏览器上运行浏览器测试。我们在我们中使用环境变量.travis.yml将构建拆分为多个作业，然后使用所需的浏览器传递到Sauce Labs期望的能力。在特拉维斯CI方面，它最终看起来像这个。

使用xvfb运行需要GUI的测试 #

要在Travis CI上运行需要图形用户界面的测试，请使用xvfb（X

Virtual Framebuffer）模仿显示器。如果您需要浏览器，则可以使用

Firefox（使用预安装的版本，或者添加在）

或谷歌Chrome（与添加在，在Linux Trusty或macOS上）。

运用services: #

这仅适用于Ubuntu 16.04（Xenial），即dist: xenial

以下将启动xvfb并为其设置正确的值DISPLAY

环境变量：

dist: xenial
services:
  - xvfb

如果您的项目需要运行要测试的Web应用程序，则需要在运行测试之前启动它。通常使用Ruby，Node.js和基于JVM的Web服务器

提供用于运行测试套件的HTML页面。因为每个构建环境至少提供一个版本的Ruby，Node.js和OpenJDK，所以您可以依赖其中一个版本

三种选择。

添加一个before_script启动服务器，例如：

before_script:
  - "export DISPLAY=:99.0"
  - "sh -e /etc/init.d/xvfb start"
  - sleep 3 # give xvfb some time to start
  - rackup  # start a Web server
  - sleep 3 # give Web server some time to bind to sockets, etc

{：data-file =“。travis.yml”}

如果您需要Web服务器在端口80上进行侦听，请记住使用sudo（Linux不允许非特权进程绑定到端口80）。对于大于1024的端口，使用sudo没有必要（不推荐）。

Note that sudo is not available for builds that are running on the container-based workers.

使用Chrome插件### 在无头模式 #

从Linux Trusty版本57和macOS版本59开始，Google Chrome可用于“无头”

模式，适用于使用Selenium和其他工具驱动基于浏览器的测试。

截至2017-05-02，这意味着stable要么beta在Linux上构建，和beta在macOS构建上。

例如，在Linux上

dist: trusty
addons:
  chrome: stable
before_install:
  - # start your web application and listen on `localhost`
  - google-chrome-stable --headless --disable-gpu --remote-debugging-port=9222 http://localhost &
  ⋮

{：data-file =“。travis.yml”}

在macOS上：

language: objective-c
addons:
  chrome: beta
before_install:
  - # start your web application and listen on `localhost`
  - "/Applications/Google\\ Chrome.app/Contents/MacOS/Google\\ Chrome --headless --disable-gpu --remote-debugging-port=9222 http://localhost &"
  ⋮

{：data-file =“。travis.yml”}

使用Firefox插件### 在无头模式 #

从版本56开始，Firefox可以在“无头”模式下使用，即

适合使用Selenium和其他工具驱动基于浏览器的测试。

可以使用。启用无头模式MOZ_HEADLESS

环境变量：

env:
  - MOZ_HEADLESS=1
addons:
  firefox: latest

{：data-file =“。travis.yml”}

或者，你可以通过-headless命令行参数何时

启动Firefox。例如，以下代码演示了您的操作方式

使用Selenium的Python客户端设置此参数：

from selenium.webdriver import Firefox
from selenium.webdriver.firefox.options import Options

options = Options()
options.add_argument('-headless')
firefox = Firefox(firefox_options=options)

使用PhantomJS #

PhantomJS是一个使用JavaScript API的无头WebKit。它是快速无头测试，站点抓取，页面捕获，SVG渲染器，网络监控和许多其他用例的最佳解决方案。

CI环境提供PhantomJS预安装（在PATH中可用phantomjs;不要依赖于确切的位置）。由于它完全无头，所以不需要运行xvfb。

一个非常简单的例子：

script: phantomjs testrunner.js

{：data-file =“。travis.yml”}

如果您需要Web服务器来提供测试，请参阅上一节。

红宝石 #

RSpec，茉莉，黄瓜 #

这是一个运行Rspec，Jasmine和Cucumber测试的示例rake任务：

task :travis do
  ["rspec spec", "rake jasmine:ci", "rake cucumber"].each do |cmd|
    puts "Starting to run #{cmd}..."
    system("export DISPLAY=:99.0 && bundle exec #{cmd}")
    raise "#{cmd} failed!" unless $?.exitstatus == 0
  end
end

在这个例子中，Jasmine和Cucumber都需要显示端口，因为它们都使用真正的浏览器。 Rspec会在没有它的情况下运行，但设置它并没有坏处。

故障排除 #

Selenium和Firefox弹出窗口 #

例如，如果您的测试套件处理模式对话框弹出窗口，重定向到另一个位置，然后您可能需要添加自定义配置文件，以便弹出窗口被抑制。

这可以通过应用关闭选项的自定义Firefox配置文件来修复:(示例在Ruby中使用Capybara）

Capybara.register_driver :selenium do |app|

  custom_profile = Selenium::WebDriver::Firefox::Profile.new

  # Turn off the super annoying popup!
  custom_profile["network.http.prompt-temp-redirect"] = false

  Capybara::Selenium::Driver.new(app, :browser => :firefox, :profile => custom_profile)
end

Karma和Firefox不活动超时 #

使用Karma和Firefox进行测试时，由于浏览器不活动超时，您可能会遇到构建错误。发生这种情况时，Karma将输出类似于以下内容的错误：

WARN [Firefox 31.0.0 (Linux)]: Disconnected (1 times), because no message in 10000 ms.

在这种情况下，您应该将浏览器不活动超时增加到更高的值karma.conf.js，例如：

browserNoActivityTimeout: 30000,

有关更多信息，请参阅Karma配置文件文档。
