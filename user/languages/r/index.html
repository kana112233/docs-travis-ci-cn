本指南涵盖的内容 #

本指南介绍了特定于R的构建环境和配置主题

项目。

请务必阅读我们的教程和构建配置首先指导。

社区支持的警告 #

Travis CI对R的支持由社区提供，可能会被删除或

随时改变。如果您遇到任何问题，请在报告中报告

Travis CI问题跟踪器和cc@jeroen

和@jimhester。

基本配置 #

Travis CI中的R支持旨在使其易于测试[[R

包](http://cran.r-project.org/doc/manuals/R-exts.html)。如果您的R包不需要任何系统依赖性

超出你的指定DESCRIPTION文件，你的.travis.yml能够

只是

language: r

{：data-file =“。travis.yml”}

R环境随之而来胶乳和pandoc

预先安装，使其更容易使用RMarkdown要么

knitr{：data-proofer-ignore =“”}

配置选项 #

Travis CI支持R包的许多配置选项。

R版本 #

Travis CI支持R版本3.0.3以及Linux Precise，Trusty和macOS。

每个主要版本都存在别名，例如3.1指着3.1.3。除此之外

名称oldrel是别名的前一个主要版本和release是别名的

最新的次要版本。devel是建立在R git镜子

R SVN主干（每小时更新）。

矩阵构建是支持R版本，但是两个实例都支持r必须

在…小写。

language: r
r:
  - oldrel
  - release
  - devel

{：data-file =“。travis.yml”}

随着新的次要版本的发布，别名将浮动并指向最多

目前的次要释放。

你可以访问上面的“频道”串（即release）而不是具体的版本号（即3.2.1）与环境变量TRAVIS_R_VERSION_STRING。

用于每个构建的确切R版本包含在“R会话信息”中

###

DESCRIPTION文件，并从CRAN安装它们。你可以包括

通过在发布中列出它们对开发中的包的依赖性Remotes:

DESCRIPTION。见遥控小插图

.travis.yml。

LaTeX / TexLive包 #

包含的TexLive分发仅包含一组有限的默认值

包。如果您的插图需要额外的TexLive套餐

可以使用安装它们tlmgr install在里面before_install步。

language: r

before_install:
  - tlmgr install index

找出你可能需要的软件包的最好方法是查看

LaTeX错误消息中列出的软件包并在其上搜索它们CTAN。

Contained in:表示包组的字段

你需要安装。

latex: false。

###

安装的默认pandoc版本是2.2。替代[pandoc

发布](https://github.com/jgm/pandoc/releases)可以通过设置安装pandoc_version到了

所需的版本。

language: r
pandoc_version: 1.19.2.1

{：data-file =“。travis.yml”}

如果您不需要Pandoc，请告诉Travis CI不要使用它pandoc: false。

APT包 #

使用APT插件安装APT包。

下面的代码片段为R包安装了先决条件xml2：

addons:
  apt:
    packages:
      - libxml2-dev

{：data-file =“。travis.yml”}

在非Linux版本中忽略此选项。

另一种选择是apt_packages键：

apt_packages:
  - libxml2-dev

{：data-file =“。travis.yml”}

包裹检查选项 #

您可以使用以下顶级选项来控制使用的选项

在构建和检查包裹时：

  • warnings_are_errors：此选项强制所有WARNINGS从R CMD check至

成为构建失败（默认true）。这在准备时特别有用

您的包裹提交给CRAN，建议用于大多数包裹。

组warnings_are_errors: false如果你不想要WARNINGS失败的构建。

  • r_build_args：传递给的其他参数R CMD build，作为单身

串。默认为空。

  • r_check_args：传递给的其他参数R CMD check，作为单身

串。默认为--as-cran。

Bioconductor的 #

一个典型的Bioconductor的包应该只需要指定

Bioconductor版本，他们希望在他们的.travis.yml。

language: r
r: bioc-devel

{：data-file =“。travis.yml”}

或者，如果您要针对发布分支进行测试

language: r
r: bioc-release

{：data-file =“。travis.yml”}

Travis CI将为该版本的Bioconductor使用适当的R版本

适当地配置Bioconductor以安装依赖项。

Packrat #

如果您希望Travis CI使用您的项目特定的packrat包库，

而不是从CRAN下载包依赖项的默认行为，您可以将其添加到您的.travis.yml：

install:
  - R -e "0" --args --bootstrap-packrat
  - R -e "packrat::restore(restart = FALSE)"

{：data-file =“。travis.yml”}

您可以通过以下方式缓存packrat包来最小化构建时间：

cache:
  directories:
    - $TRAVIS_BUILD_DIR/packrat/src
    - $TRAVIS_BUILD_DIR/packrat/lib
  packages: true

{：data-file =“。travis.yml”}

杂 #

  • cran：CRAN镜像用于获取包。默认为

https://cloud.r-project.org。

  • repos：要传递给的存储库字典options(repos)。如果CRAN是

没有在字典中给出的价值cran使用选项。

例：

repos:
  CRAN: https://cloud.r-project.org
  ropensci: http://packages.ropensci.org

{：data-file =“。travis.yml”}

  • disable_homebrew：如果true这删除了预装的自制软件

在macOS上安装。用于测试包是否构建在vanilla macOS上

机器，例如CRAN mac builder。

环境变量 #

R-Travis设置了以下附加环境变量[特拉维斯

默认](/user/environment-variables/#default-environment-variables)。

  • TRAVIS_R_VERSION=3.2.4设置为选择的版本r:。
  • R_LIBS_USER=~/R/Library
  • R_LIBS_SITE=/usr/local/lib/R/site-library:/usr/lib/R/site-library
  • _R_CHECK_CRAN_INCOMING_=false
  • NOT_CRAN=true
  • R_PROFILE=~/.Rprofile.site
  • TRAVIS_R_VERSION_STRING设置为串提供至r:，即release，oldrel要么devel。

例如，仅用于部署有用release通过on: condition: "$TRAVIS_R_VERSION_STRING = release"。

其他依赖字段 #

对于大多数软件包，您不需要在其中指定任何其他依赖项

您的.travis.yml。但是对于极少数情况下面的字段

得到支持。

以下每个名称都是您可以选择指定为的包列表

您的顶级条目.travis.yml;这些列表中的条目将是

  • apt_packages
  • brew_packages：要安装的软件包列表brew。这个选项是

在非macOS版本上被忽略。

  • r_binary_packages：要作为二进制包安装的R包列表

linux通过Michael Rutter建立

cran2deb4ubuntu PPA。

这些安装将比源安装更快，但可能并不总是如此

最新版本。像安装时一样指定名称

来自CRAN。在macOS版本中，这些包是从源安装的。

  • r_packagesinstall.packages。
  • bioc_packagesBioconductor的

要安装的软件包。

  • r_github_packages：直接从GitHub安装的软件包列表，

运用devtools::install_github来自

devtools包。包名称

这应该是形式user/repo。

如果软件包安装在子目录中，请使用user/repo/subdirectory

user/repo要么user/repo/folder至

该Remotes部分DESCRIPTION你的包裹的文件

自定义Travis构建步骤 #

对于某些高级用例，覆盖使用的默认步骤是有意义的

用于构建R包。默认规则大致相当于：

install:
- R -e 'devtools::install_deps(dep = T)'

script:
- R CMD build .
- R CMD check *tar.gz

{：data-file =“。travis.yml”}

如果您想查看完整的详细信息，请参阅

源代码。

例子 #

###

如果您的包位于存储库的子目录中，则需要

在运行之前更改到子目录install要么script脚步。

language: r
before_install:
  - cd subdirectory

{：data-file =“。travis.yml”}

远程包 #

如果您的包依赖于您可以使用的另一个存储库r_github_packages通过这种方式：

r_github_packages: user/repo

{：data-file =“。travis.yml”}

另一种方法是将以下行添加到您的DESCRIPTION

Imports: pkg-name-of-repo
Remotes: user/repo

{：data-file =“DESCRIPTION”}

记住这一点Remotes:指定资源一个开发包，所以仍然需要列出包Imports:，Suggests:Depends:要么LinkingTo:。

在极少数情况下回购和名字不同，Remotes:期待知识库Imports:期待包名称（按照DESCRIPTION那个进口包装）。

远程包在子目录中 #

如果您的包依赖于将包保存在子目录中的另一个存储库，则可以使用r_github_packages通过这种方式：

r_github_packages: user/repo/folder

{：data-file =“。travis.yml”}

DESCRIPTION

Remotes: user/repo/folder

{：data-file =“DESCRIPTION”}
