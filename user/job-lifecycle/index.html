Job Lifecycle

  • 工作生命周期
  • 自定义安装阶段
  • 自定义构建阶段

Travis CI为每种编程语言提供默认构建环境和默认的阶段集。使用您的作业的构建环境创建虚拟机，将存储库克隆到其中，安装可选的插件，然后运行构建阶段。

继续阅读，了解如何通过您的方式自定义此过程中的任何阶段.travis.yml。

工作生命周期 #

  1. 安装：安装所需的任何依赖项
  2. 脚本：运行构建脚本

您可以在安装阶段之前运行自定义命令（before_install）之前（before_script）或之后（after_script）脚本阶段。

当您的构建成功或失败时，您可以执行其他操作after_success（如建筑文件）或after_failure（例如上传日志文件）阶段。

同时after_failure和after_success，您可以使用。访问构建结果$TRAVIS_TEST_RESULT环境变量。

完整的作业生命周期，包括三个可选的部署阶段，以及在检出git存储库和更改到存储库目录之后，是：

  1. 可选安装apt addons
  2. 可选安装cache components
  3. before_install
  4. install
  5. before_script
  6. script
  7. 可选的before_cache（用于清理缓存）
  8. after_success要么after_failure
  9. 可选的before_deploy
  10. 可选的deploy
  11. 可选的after_deploy
  12. after_script

一个建立可以由许多工作组成。

自定义安装阶段 #

默认依赖项安装命令取决于项目语言。

例如，Java构建使用Maven或Gradle，具体取决于存储库中存在的构建文件。当存储库中存在Gemfile时，Ruby项目使用Bundler。

您可以指定自己的脚本来安装项目依赖项：

install: ./install-dependencies.sh

{：data-file =“。travis.yml”}

使用自定义脚本时，它们应该是可执行的（例如，使用chmod +x并且包含有效的shebang线，例如/usr/bin/env sh，/usr/bin/env ruby， 要么/usr/bin/env python。

您还可以提供多个步骤，例如安装ruby和节点依赖项：

install:
  - bundle install --path vendor/bundle
  - npm install

{：data-file =“。travis.yml”}

当安装中的某个步骤失败时，构建会立即停止并标记为出错。

你也可以使用apt-get要么snap至安装依赖项

跳过安装阶段 #

完全通过将以下内容添加到您的安装步骤来跳过安装步骤.travis.yml：

install: true

{：data-file =“。travis.yml”}

自定义构建阶段 #

默认构建命令取决于项目语言。 Ruby项目使用rake，大多数Ruby项目的共同点。

您可以覆盖默认构建步骤.travis.yml：

script: bundle exec thor build

{：data-file =“。travis.yml”}

您也可以指定多个脚本命令：

script:
- bundle exec rake build
- bundle exec rake builddoc

{：data-file =“。travis.yml”}

当其中一个构建命令返回非零退出代码时，Travis CI构建也会运行后续命令，并累积构建结果。

在上面的例子中，如果bundle exec rake build返回退出代码1，以下命令bundle exec rake builddoc仍在运行，但构建将导致失败。

如果您的第一步是运行单元测试，然后进行集成测试，您可能仍希望在单元测试失败时查看集成测试是否成功。

您可以通过使用一点shell魔术来随后运行所有命令来更改此行为，但是当第一个命令返回非零退出代码时仍然会使构建失败。这是你的片段.travis.yml

script: bundle exec rake build && bundle exec rake builddoc

{：data-file =“。travis.yml”}

这个例子（注意&&）立即失败bundle exec rake build失败。

注意$？ #

每个命令都在script由特殊的bash函数处理。

这个功能操纵$?生产适合展示的原木。

因此，你不应该依赖于它的价值$?在script部分到

改变构建行为。

复杂构建命令 #

如果您有一个很难配置的复杂构建环境.travis.yml，考虑将步骤移动到单独的shell脚本中。

该脚本可以是您的存储库的一部分，可以从中轻松调用.travis.yml。

考虑一种情况，您希望运行更复杂的测试方案，但仅适用于不是来自pull请求的构建。 shell脚本可能是：

#!/bin/bash
set -ev
bundle exec rake:units
if [ "${TRAVIS_PULL_REQUEST}" = "false" ]; then
  bundle exec rake test:integration
fi

请注意set -ev在顶部。该-e只要一个命令返回非零退出代码，flag就会导致脚本退出。如果您想要任何脚本提前退出，这可能很方便。它还有助于复杂的安装脚本，其中一个失败的命令不会导致安装失败。

该-vflag使shell在执行脚本之前打印所有行，这有助于识别哪些步骤失败。

从你的脚本运行该脚本.travis.yml：

  1. 将其保存在您的存储库中scripts/run-tests.sh。
  2. 通过运行使其可执行chmod ugo+x scripts/run-tests.sh。
  3. 将其提交到您的存储库。
  4. 把它添加到你的.travis.yml： ``` script: ./scripts/run-tests.sh

{：data-file =“。travis.yml”}

#### 这是如何运作的？ （或者，为什么你不应该使用`exit`#### 在构建步骤中）

作业生命周期中指定的步骤将编译为单个bash脚本并在worker上执行。

覆盖这些步骤时，请勿使用`exit`shell内置命令。

这样做会冒着终止构建过程而不给Travis一个机会的风险

执行后续任务。

运用`exit`在构建期间将调用的自定义脚本中的内容很好。

## 打破构建

如果作业生命周期的前四个阶段中的任何命令返回非零退出代码，则构建将被破坏：

- `before_install``install``before_script`返回非零退出代码，

**出错**并立即停止。
- `script`返回一个非零退出代码，构建是****，但在被标记为之前继续运行**失败**

退出代码`after_success`，`after_failure`，`after_script`，`after_deploy`和后续阶段不会影响构建结果。
