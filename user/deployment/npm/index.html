Travis CI可以自动发布您的npm包npmjs.com

或者在成功构建之后的另一个类似npm的注册表。默认情况下Travis CI

发布到npmjs.com，但是如果你有publishConfig.registry关键在你的

package.json然后Travis CI发布到该注册表。

最小的.travis.yml发布到的配置npmjs.com与npm版本2+看起来像：

deploy:
  provider: npm
  email: "YOUR_EMAIL_ADDRESS"
  api_key: "YOUR_AUTH_TOKEN"

{：data-file =“。travis.yml”}

你可以拥有travis工具为您设置一切：

$ travis setup npm

请记住，上面的命令必须在项目目录中运行，所以

它可以修改.travis.yml为了你。

NPM身份验证令牌 #

您的NPM Auth Token可以通过以下方式获得：

  1. 登录您的NPM帐户，并在以下位置生成新令牌https://www.npmjs.com/settings/USER/tokens，哪里

USER是能够发布npm包的用户帐户的名称。

  1. 使用NPM CLI命令npm adduser创建一个用户，然后打开~/.npmrc文件：
    1. 对于NPM v2 +，请使用authToken值。
    2. 对于NPM~1，请使用auth值。

总是加密你的身份验证令牌。假设您安装了Travis CI命令行客户端，您可以这样做：

$ travis encrypt YOUR_AUTH_TOKEN --add deploy.api_key

要发布什么 #

最有可能的是，你只想在新版本的时候部署到npm

包裹被削减。为此，您可以告诉Travis CI仅部署标记

提交，像这样：

deploy:
  ...
  on:
    tags: true

{：data-file =“。travis.yml”}

如果您在本地标记提交，请记住运行git push --tags为了确保

您的标签上传到GitHub。

您可以使用显式指定要从中释放的分支上选项：

deploy:
  ...
  on:
    branch: production

{：data-file =“。travis.yml”}

或者，您也可以将Travis CI配置为从所有分支发布：

deploy:
  ...
  on:
    all_branches: true

{：data-file =“。travis.yml”}

##

deploy:
  ...
  skip_cleanup: true

{：data-file =“。travis.yml”}

有条件的释放 #

部署问题是

多次尝试时报告。

我们建议只使用一个作业进行部署

有条件释放on:。

标记版本 #

您可以自动添加npm分发标签随着tag选项：

deploy:
  ...
  tag: next

{：data-file =“。travis.yml”}

注意.gitignore #

请注意npm部署荣誉.gitignore.npmignore

这意味着如果您的构建在其中列出的位置创建工件.gitignore，

它们不会包含在上传的包中。

看到npm文件

更多细节。

.gitignorefile匹配构建创建的东西，使用

before_deploy改变

它的内容，或创建（可能是空的）.npmignore文件

覆盖它。

在部署之前和之后运行命令 #

有时您希望在部署之前或之后运行命令。你可以使用before_deploy和after_deploy

before_deploy: "echo 'ready?'"
deploy:
  ..
after_deploy:
  - ./after_deploy_1.sh
  - ./after_deploy_2.sh

{：data-file =“。travis.yml”}

“npm ERR疑难解答！您需要付费帐户才能执行此操作。” #

范围包是

私有默认。您可以明确告诉npm您的包是一个公共包

并通过添加以下内容来避免此错误package.json

  "publishConfig": {
    "access": "public"
  },
