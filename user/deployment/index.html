上传文件和skip_cleanup #

将文件部署到提供程序时，请阻止Travis CI重置您的文件

工作目录并删除构建期间所做的所有更改（git stash --all） 通过增加skip_cleanup到你的.travis.yml：

deploy:
  skip_cleanup: true

{：data-file =“。travis.yml”}

##

通过添加不同的提供程序，可以部署到多个提供程序

到了deploy部分作为清单。例如，如果要部署到两者

cloudControl和Heroku，你的deploy部分看起来像这样：

deploy:
  - provider: cloudcontrol
    email: "YOUR CLOUDCONTROL EMAIL"
    password: "YOUR CLOUDCONTROL PASSWORD"
    deployment: "APP_NAME/DEP_NAME"
  - provider: heroku
    api_key: "YOUR HEROKU API KEY"

{：data-file =“。travis.yml”}

有条件释放on: #

通过配置，将您的构建设置为仅在特定情况下进行部署on:任何部署提供商的密钥。

deploy:
  provider: s3
  access_key_id: "YOUR AWS ACCESS KEY"
  secret_access_key: "YOUR AWS SECRET KEY"
  bucket: "S3 Bucket"
  skip_cleanup: true
  on:
    branch: release
    condition: $MY_ENV = super_awesome

{：data-file =“。travis.yml”}

什么时候所有条件中规定的条件on:部分得到满足，您的构建将部署。

使用以下选项配置条件部署：

  • repo：在表格中owner_name/repo_name。仅在特定存储库上进行构建时进行部署。例如 ``` deploy: provider: s3 on: repo: travis-ci/dpl

{：data-file =“。travis.yml”}
* `branch`：分支的名称。

如果省略，则默认为`app`特定的分支，或`master`。如果未提前知道分支名称，则可以指定

`all_branches: true`代替`branch:`并使用其他条件来控制您的部署。
* `jdk`，`node`，`perl`，`php`，`python`，`ruby`，`scala`，`go`：对于支持多个版本的语言运行时，

您可以限制部署仅在与特定版本匹配的作业上发生。
* `condition`：部署时单身bash条件评估为`true`。这必须是字符串值，相当于`if [[ <condition> ]]; then <deploy>; fi`。例如，`$CC = gcc`。
* `tags`可`true``false`或任何其他字符串：
   * `tags: true`：当且仅当是，才会触发部署`$TRAVIS_TAG`已设定。

根据您的工作流程，您可以设置`$TRAVIS_TAG`显然，即使这样

启动时的非标记构建。这导致了`branch`条件被忽略。
   * `tags: false`：当且仅当是，才会触发部署`$TRAVIS_TAG`是空的。

这也导致了`branch`条件被忽略。
   * 什么时候`tags`未设置，或设置为任何其他值，`$TRAVIS_TAG`被忽略了，而且`branch`条件被考虑，如果它被设置。


### 条件部署的示例

此示例仅部署到Appfog`staging`测试在Node.js版本0.11上运行时分支。

language: node_js deploy: provider: appfog user: … api_key: … on: branch: staging node_js: ‘0.11’ # this should be quoted; otherwise, 0.10 would not work


{：data-file =“。travis.yml”}

下一个示例使用自定义脚本进行部署`deploy.sh`，仅适用于分支机构的构建`staging`和`production`。

deploy: provider: script script: deploy.sh on: all_branches: true condition: $TRAVIS_BRANCH =~ ^staging|production$


{：data-file =“。travis.yml”}

下一个示例使用自定义脚本进行部署`deploy_production.sh`和`deploy_staging.sh`取决于触发作业的分支。

deploy:

  • provider: script script: deploy_production.sh on: branch: production
  • provider: script script: deploy_staging.sh on: branch: staging

{：data-file =“。travis.yml”}

下一个示例仅在部署时部署到S3`$CC`被设置为`gcc`。

deploy: provider: s3 access_key_id: “YOUR AWS ACCESS KEY” secret_access_key: “YOUR AWS SECRET KEY” skip_cleanup: true bucket: “S3 Bucket” on: condition: “$CC = gcc”


{：data-file =“。travis.yml”}

当设置标记且Ruby版本为2.0.0时，此示例将部署到GitHub Releases。

deploy: provider: releases api_key: “GITHUB OAUTH TOKEN” file: “FILE TO UPLOAD” skip_cleanup: true on: tags: true rvm: 2.0.0


{：data-file =“。travis.yml”}

如果您参与或试验[部署工具](https://github.com/travis-ci/dpl)确保使用GitHub的边缘版本：
