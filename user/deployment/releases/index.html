Travis CI可以自动将资产上传到GitHub存储库上的git标签。

对于最小配置，请将以下内容添加到您的.travis.yml：

deploy:
  provider: releases
  api_key: "GITHUB OAUTH TOKEN"
  file: "FILE TO UPLOAD"
  skip_cleanup: true
  on:
    tags: true

{：data-file =“。travis.yml”}

此配置将使用“GITHUB OAUTH TOKEN”上传“FILE TO UPLOAD”

（在标记的构建上相对于工作目录）。

确保你有skip_cleanup调成true否则Travis CI将删除在构建期间创建的所有文件，这可能会删除您尝试上传的内容。

GitHub Releases适用于git标签，因此非常重要

您了解标签如何影响GitHub版本。

仅在标记的构建上进行部署 #

同on.tags: true，

当且仅当构建是标记的时，您的Releases部署才会触发

建立。

定期发布 #

当。。。的时候draft选项未设置为true（详见下文），定期

发布已创建。

常规版本需要标签。

如果你设置on.tags: true（作为本文档的初始示例），这个

要求得到满足。

草案发布时间draft: true #

同

deploy:
  provider: releases
  api_key: "GITHUB OAUTH TOKEN"
  file: "FILE TO UPLOAD"
  skip_cleanup: true
  draft: true

{：data-file =“。travis.yml”}

生成的部署是一个只有存储库协作者的草稿版本

可以看到。

这使您有机会检查和编辑草稿版本。

在部署时设置标记 #

GitHub Releases在部署时需要一个标签。

而on.tags: true保证这一点，你可以推迟设置标签

您拥有所需的所有信息。

一个自然的地方是before_deploy。

例如：

    before_deploy:
      # Set up git user name and tag this commit
      - git config --local user.name "YOUR GIT USER NAME"
      - git config --local user.email "YOUR GIT USER EMAIL"
      - export TRAVIS_TAG=${TRAVIS_TAG:-$(date +'%Y%m%d%H%M%S')-$(git log --format=%h -1)}
      - git tag $TRAVIS_TAG
    deploy:
      provider: releases
      api_key: "GITHUB OAUTH TOKEN"
      file: "FILE TO UPLOAD"
      skip_cleanup: true

{：data-file =“。travis.yml”}

在部署时未设置标记时 #

如果在部署时仍未设置标记，则进行部署

provider尝试将当前提交与来自远程的标记进行匹配，

如果找到一个，使用它。

如果为当前提交分配了多个标记，则可能会出现问题

你想要的那个不匹配。

在这种情况下，请指定所需的标签（方法取决于您的使用方法）

案例）$TRAVIS_TAG解决问题。

如果构建提交在部署时与任何标记都不匹配，GitHub会创建一个

何时创建发布。

GitHub生成的标签是这种形式untagged-*，哪里*是随机的

十六进制字符

请注意，此标记在GitHub上立即可用，因此

将触发新的Travis CI构建，除非它被阻止

/^untagged/

覆盖该版本上的现有文件 #

如果需要覆盖现有文件，请添加overwrite: truedeploy.travis.yml。

使用Travis CI客户端填充初始部署配置 #

你也可以使用Travis CI命令行客户端配置你的.travis.yml：

travis setup releases

或者，如果您使用的是私有存储库或GitHub Apps集成：

travis setup releases --com

使用OAuth令牌进行身份验证 #

public_reporepo上传资产的范围。强烈建议使用，而不是手动设置travis setup releases

这导致类似于：

deploy:
  provider: releases
  api_key:
    secure: YOUR_API_KEY_ENCRYPTED
  file: "FILE TO UPLOAD"
  skip_cleanup: true
  on:
    tags: true

{：data-file =“。travis.yml”}

警告：该public_reporepoGitHub oauth令牌的范围授予对所有用户（公共）存储库的写访问权。为了安全起见，它是理想的选择api_key机器用户

##

userpassword选项。建议不要这样做，因为它允许完全访问您的GitHub帐户，但最简单的设置。建议使用加密密码travis encrypt "GITHUB PASSWORD" --add deploy.password。此示例使用用户名和密码进行身份验证。

deploy:
  provider: releases
  user: "GITHUB USERNAME"
  password: "GITHUB PASSWORD"
  file: "FILE TO UPLOAD"
  skip_cleanup: true
  on:
    tags: true

部署到GitHub Enterprise #

如果您希望将资源上载到GitHub Enterprise存储库，则必须覆盖$OCTOKIT_API_ENDPOINT您的GitHub Enterprise API端点的环境变量：

http(s)://"GITHUB ENTERPRISE HOSTNAME"/api/v3/

你可以配置它存储库设置或通过你的.travis.yml：

env:
  global:
    - OCTOKIT_API_ENDPOINT="GITHUB ENTERPRISE API ENDPOINT"

{：data-file =“。travis.yml”}

上传多个文件 #

您可以使用yml数组表示法上载多个文件。此示例上载两个文件。

deploy:
  provider: releases
  api_key:
    secure: YOUR_API_KEY_ENCRYPTED
  file:
    - "FILE 1"
    - "FILE 2"
  skip_cleanup: true
  on:
    tags: true

{：data-file =“。travis.yml”}

您还可以通过设置启用通配符file_glob至true。这个例子

包括给定目录中的所有文件。

deploy:
  provider: releases
  api_key: "GITHUB OAUTH TOKEN"
  file_glob: true
  file: directory/*
  skip_cleanup: true
  on:
    tags: true

{：data-file =“。travis.yml”}

您可以使用glob模式递归查找文件：

deploy:
  provider: releases
  api_key: "GITHUB OAUTH TOKEN"
  file_glob: true
  file: directory/**/*
  skip_cleanup: true
  on:
    tags: true

{：data-file =“。travis.yml”}

请注意所有路径file是相对于当前工作目录，而不是$TRAVIS_BUILD_DIR。

有条件的释放 #

只有在满足某些条件时才能部署。

看到有条件释放on:。

在发布之前或之后运行命令 #

有时您希望在发布gem之前或之后运行命令。你可以使用before_deploy和after_deploy这个阶段。只有在特拉维斯CI实际推动发布时才会触发这些。

before_deploy: "echo 'ready?'"
deploy:
  ..
after_deploy:
  - ./after_deploy_1.sh
  - ./after_deploy_2.sh

{：data-file =“。travis.yml”}

高级选项 #

选项来自.travis.yml被传递到Octokit API

#create_release

和#update_release方法，

所以你可以使用任何有效的Octokit选项，

除非在本文件中单独处理。

这些包括：

  • name
  • body
  • prerelease（布尔值）

请注意格式化body是没有保留。
