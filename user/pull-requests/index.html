Building Pull Requests

  • 拉动请求的“双重构建”

拉取请求构建是Travis CI的重要组成部分。

每当在GitHub上打开拉取请求时，Travis CI都会构建它并更新拉取请求页面上的状态图标。

请注意，草案请求Travis CI目前不支持。

如何建立拉请求 #

当在GitHub上打开拉取请求时，Travis CI会收到通知并运行构建。

在构建期间，我们将拉取请求的状态图标更新为以下状态之一：

  • 警告构建仍在运行。
  • 构建失败的通知 - 不应合并拉取请求。

Travis CI在首次打开时构建一个pull请求，并且只要将提交添加到pull请求中。

而不是构建已推送到分支的提交，拉请求来自，我们在源分支和上游分支之间构建合并。

要仅构建不在拉取请求上的推送事件，请禁用建立在拉动请求上在您的存储库设置中。

请求和安全限制 #

拉取请求最重要的限制是安全环境变量和加密数据。

可以操纵从上游存储库的分支发送的拉取请求以暴露环境变量。

上游存储库的维护者无法抵御此攻击，因为任何在GitHub上分配存储库的人都可以发送拉取请求。

Travis CI使加密变量和数据仅用于来自同一存储库的拉取请求。这些被认为是值得信赖的，因为只有对存储库具有写访问权限的成员才能发送它们。

从分叉存储库发送的拉取请求无法访问加密变量或数据，即使这些是在fork源项目中定义的。

如果您的构建依赖加密变量来运行，例如运行Selenium测试BrowserStack

这些测试来自外部贡献者的拉取请求。

要解决此问题，请将这些测试仅限于以下情况

环境变量可用，或者完全禁用它们用于拉取请求，如以下示例所示：

script:
   - 'if [ "$TRAVIS_PULL_REQUEST" != "false" ]; then bash ./travis/run_on_pull_requests; fi'
   - 'if [ "$TRAVIS_PULL_REQUEST" = "false" ]; then bash ./travis/run_on_non_pull_requests; fi'

{：data-file =“。travis.yml”}

我的Pull请求尚未构建 #

如果未构建拉取请求或未在Travis CI的用户界面中显示拉取请求，则通常意味着无法合并。

我们依赖于GitHub在源分支的变化和发送pull请求的上游分支之间透明地创建的合并提交。

因此，当您创建或更新拉取请求时，Travis CI不会创建

为它构建，确保pull请求是可合并的。

当上游分支更新时，Travis CI当前也不构建拉取请求，因为这会导致过多的新构建。

如果拉取请求已合并，则无法重新运行该作业。你会得到一个错误：

The command "eval git fetch origin +refs/pull/994/merge: " failed

恢复合并拉取请求的分支不会触发构建，也不会将新提交推送到已合并的分支。

拉动请求的“双重构建” #

如果在GitHub pull请求中看到两个构建状态图标，则表示该分支有一个构建，并且一个构建用于pull请求本身（实际上是head分支与pull中指定的基本分支的合并构建请求）。

建立推动分支和构建推拉请求控制这种行为。
