Speeding up the build

Travis CI实现了一些有助于加速构建的优化，

就像在DB文件的内存文件系统中一样，但是有很多东西

可以做更多的事情来改善构建时间。

跨虚拟机并行构建 #

为了加速测试套件，您可以使用它将其分解为几个部分

特拉维斯CI[建立

矩阵](/user/customizing-the-build/#build-matrix)

特征。

假设您要将单元测试和集成测试分成两部分

不同的构建工作。它们将并行运行并充分利用可用的

为您的帐户建立容量。

以下是如何在您的网站中使用此功能的示例.travis.yml：

env:
  - TEST_SUITE=units
  - TEST_SUITE=integration

{：data-file =“。travis.yml”}

然后更改脚本命令以使用新的环境变量

确定要运行的脚本。

script: "bundle exec rake test:$TEST_SUITE"

{：data-file =“。travis.yml”}

Travis CI将根据环境变量确定构建矩阵

安排两个版本运行。

关于这个设置的整洁部分是单元测试套件通常会发生的

在集成测试套件之前完成，为您提供更快的视觉反馈

在基本测试覆盖范围内。

进一步。您可以将集成测试的不同问题分开

不同的子文件夹，并在构建矩阵的不同阶段运行它们。

env:
  - TESTFOLDER=integration/user
  - TESTFOLDER=integration/shopping_cart
  - TESTFOLDER=integration/payments
  - TESTFOLDER=units

{：data-file =“。travis.yml”}

然后，您可以调整脚本命令以为每个子文件夹运行rspec：

script: "bundle exec rspec $TESTFOLDER"

{：data-file =“。travis.yml”}

例如，Rails项目使用构建矩阵功能来创建单独的

每个要测试的数据库的作业，以及通过分割测试

关心。一套仅针对铁路运行测试，另一套针对动作包进行测试，

actionmailer，activesupport和一大堆集合运行activerecord

测试多个数据库。看他们的[.travis.yml

文件](https://github.com/rails/rails/blob/master/.travis.yml)更多例子。

在一个虚拟机上并行构建 #

在一个虚拟机上并行化测试套件取决于语言和测试运行器：

  • 对于Ruby和RSpec使用parallel_tests
  • 对于Java，请使用内置功能[并行运行测试

使用JUnit](http://incodewetrustinc.blogspot.com/2009/07/run-your-junit-tests-in-parallel-with.html)。

为了让您了解加速，我们正在尝试并行运行测试travis-core在4个工作岗位上，我能看到从大约26分钟到大约19分钟的下降。

在多个VM上并行化RSpec，Cucumber和Minitest #

如果您想在多个VM上对RSpec，Cucumber或Minitest进行并行测试，以便从CI获得更快的反馈，那么您可以尝试背包宝石。它将跨虚拟机分割测试，并确保测试将在每个VM上运行相当的时间（每个作业将花费相似的时间）。您可以使用我们的矩阵功能来设置背包。

RSpec并行化示例 #

script: "bundle exec rake knapsack:rspec"
env:
  global:
    - MY_GLOBAL_VAR=123
    - CI_NODE_TOTAL=2
  matrix:
    - CI_NODE_INDEX=0
    - CI_NODE_INDEX=1

{：data-file =“。travis.yml”}

这样的配置将生成具有2个以下ENV行的矩阵：

MY_GLOBAL_VAR=123 CI_NODE_TOTAL=2 CI_NODE_INDEX=0
MY_GLOBAL_VAR=123 CI_NODE_TOTAL=2 CI_NODE_INDEX=1

黄瓜并行化的例子 #

script: "bundle exec rake knapsack:cucumber"
env:
  global:
    - CI_NODE_TOTAL=2
  matrix:
    - CI_NODE_INDEX=0
    - CI_NODE_INDEX=1

{：data-file =“。travis.yml”}

最小的并行化示例 #

script: "bundle exec rake knapsack:minitest"
env:
  global:
    - CI_NODE_TOTAL=2
  matrix:
    - CI_NODE_INDEX=0
    - CI_NODE_INDEX=1

{：data-file =“。travis.yml”}

RSpec，Cucumber和Minitest并行化示例 #

如果要同时为RSpec，Cucumber和Minitest并行化测试套件，则在其中定义脚本.travis.yml这条路：

script:
  - "bundle exec rake knapsack:rspec"
  - "bundle exec rake knapsack:cucumber"
  - "bundle exec rake knapsack:minitest"

{：data-file =“。travis.yml”}

您可以在中找到更多示例背包文档。

缓存依赖项 #

为项目安装依赖项可能需要相当长的时间才能完成更大的项目。在

为了使速度更快，您可以尝试缓存依赖项。

你可以使用我们的内置缓存或者在S3上滚动你自己。如果你

想要自己滚动，你使用Ruby与Bundler，看看伟大的WAD项目。

对于其他语言，您可以直接使用s3工具上载和下载依赖项。

特定于环境的加速构建的方法 #

除了特拉维斯实施的优化之外，还有

您可以考虑采用几种特定于环境的方法来提高速度

你的考试。

PHP优化 #

Travis CI上的PHP VM映像提供了几个PHP版本，其中包括

XDebug的。如果您希望生成代码覆盖率，则XDebug扩展非常有用

您的Travis版本中的报告，但已被证明具有负面影响

表演。

你可以考虑一下

禁用PHP XDebug扩展为您

建立如果：

  • 您没有在Travis测试中生成代码覆盖率报告;要么
  • 您正在测试PHP 7.0或更高版本，并能够使用PHP调试器（phpdbg）

这可能会更快。

使用phpdbg示例 #

before_script:
  - phpenv config-rm xdebug.ini
  - composer install

script:
  - phpdbg -qrr phpunit

{：data-file =“。travis.yml”}

Makefile优化 #

如果您的makefile构建包含可以安全的独立部分

并行化，你可以[运行多个食谱

同时](https://www.gnu.org/software/make/manual/html_node/Parallel.html)。

看到[虚拟化

环境](/user/reference/overview/#virtualization-environments)确定

一个环境通常有多少CPU并设置make作业参数到

相似的数字（如果您的构建经常在磁盘I / O上等待，则略高一些）。

请注意，这样做会导致并发配方输出交错。

Makefile并行化示例 #

env:
  global:
    - MAKEFLAGS="-j 2"

{：data-file =“。travis.yml”}
